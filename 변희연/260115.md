### Git

**Branch**

- 독립적인 개발 공간
- 병렬 개발로 협업에 유리
- 작업 기록에도 유리

**Git flow : branch 전략**

- -f , force 금지
- **master(main)** : 사용자에게 배포
- **release** : 배포 전 최종 버그 수정 및 점검
- **develop** : 다음 버전 출시를 위해 개발 진행
- **feature** : 새로운 기능을 개발
- **hotfix** : 배포 중인 서비스(master)에서 급한 버그 수정

**간소화된 Git Flow 워크플로우**

1. develop 브랜치 생성 : (master) ...$ git checkout -b develop
2. feature 브랜치 생성 : (develop) ...$ git checkout -b fe-feature/login
3. 기능 개발 진행 : feature 브랜치에서 개발 진행
4. develop 브랜치로 Merge Request(MR) : 개발 내용을 develop 브랜치로 머지
5. master 브랜치로 Merge Request(MR) : 배포 가능할 정도로 개발이 완료 되면 develop 브랜치에서 master 브랜치로 Merge

**Release와 Hotfix Branch**

→ release branch와 hotfix branch는 프로젝트 규모에 따라 사용.

1. release Branch
    - 다음 버전 개발과 현재 버전 배포 준비를 동시에 할 때
    - 앱스토어 심사, 정기적인 배포가 필요한 경우
2. hotfix Branch
    - 현재 배포 중인 버전에서 버그가 발생하여 급하게 적용해야 할때

**알아보면 좋은 것**

- MR 템플릿 : Merge Request
- Commit Message 템플릿
- Branch / Commit 제목 / MR 제목 네이밍 컨벤션
    - Branch : 자세한 작업 내용
    - feat : 작업한 내용
    - [Feat] 이거 개발했어요
- Commit Message 컨벤션
    - feat, fix, refact, …
    - 디렉토리 구조
