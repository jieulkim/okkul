## Today I Learned

### DB 설계 및 반정규화

**적절한 데이터 타입 선택**

- 데이터 유형은 일반적으로 문자형 / 숫자형 / 날짜형으로 구분
- 같은 문자, 숫자 안에서도 다양한 데이터 타입이 있으니 저장 용량을 고려해서 선택
    - 저장 데이터에 비해 과하지 않게 데이터 타입을 설정해야 용량 낭비가 적어짐
- 일반적으로 데이터를 저장하는데 문제가 없는 타입 중 가장 작은 것을 선택
    - 작은 데이터 타입일수록 디스크나 메모리에 더 적은 공간을 사용하므로 더 빠름

**DB 정규화 특징**

- 테이블 간에 중복된 데이터를 최소화
    - 저장 용량의 최적화 가능
    - 데이터의 무결성 유지
    - 쿼리 내의 Join을 많이 사용하게 되어 응답시간 저하를 초래할 수 있음

**DB 반 정규화란?**

- 시스템의 성능 향상, 개발 과정의 편의성, 운영의 단순화를 위해
- 정규화 원칙에 위배되는 행위를 의도적으로 수행하게 되는 일련의 과정
    - 정규화 이후에 수행하는 것 (반정규화부터 수행하는 것은 X)

**DB 반정규화 고려 대상**

- 대량의 범위를 자주 처리해야 하는 경우
- 특정 범위의 데이터만 자주 처리하는 경우
- 요약/집계 정보가 자주 요구되는 경우
- Join으로 인한 성능 저하가 예상될 때

**반정규화**

- 1:1 관계의 테이블 병합
- 1:N 관계의 테이블 병합
- 슈퍼 / 서브 타입 테이블 병합
- 수직분할 (집중화된 일부 컬럼을 분리)
- 수평분할 (행으로 구분하여 구간별 분리)
- 테이블 추가 (중복테이블, 통계테이블, 이력테이블, 부분테이블)

### Jira 사용법

**왜 Jira를 사용해야 하는가?**

- 애자일 모델
    - 중간에 수정하고 돌아가기 쉬움
    - 변화가 생길 시 대처가 용이
    - 개발 중간 완성본에 대한 확인이 가능함
    - 스프린트 (일정 기간 동안 팀이 정해진 작업을 완료하는 짧고 반복적인 개발 주기)를 반복
- 애자일 모델 실현 도구 - 칸반보드와 스크럼
    - 칸반보드 : 스프린트 시작 전 해당 기간 동안 할 일을 백로그에 모두 작성
    - 스크럼 : 활성 스프린트를 기반 진행 상황 공유 및 계획 수정

**Jira 프로젝트 구조**

- 스프린트 (1주)
    - 일정량의 작업을 완료하는 월요일 ~ 금요일(또는 일요일)까지의 고정된 기간 (1주)
- 에픽 (Epic)
    - 완료까지 여러 번의 스프린트가 요구되는 큰 업무 단위
    - 스프린트에 에픽 추가 X → 이슈 추가 O
        - ex) 유저 인증, 프론트 개발, 백엔드 개발
    
    ⇒ Epic의 규모가 너무 크면 Story의 규모도 너무 커질 수 있음
    

- 스토리 (Story)
    - 사용자 관점에서 작성된 요구 사항
        - ex) 유저는 본인 계정에 접근할 수 있어야 한다.

- 작업 (Task) - 개발
    - 스토리를 완료하기 위해 수행해야 하는 구체적인 활동
        - ex) 로그인 화면 UI 구현, 로그인 인증 API 개발

- 하위 작업 (Sub Task)
    - 스토리/작업의 하위 작업으로 더 작은 단위로 나눈 것
        - ex) 로그인 UI XML 작업, 로그인 API 연결
    - 유의사항
        - Jira의 보드가 다소 복잡하게 표현될 수 있음
        - 하위 작업 스토리 포인트는 번다운 차트에 반영되지 않음

⇒ 분류 기준은 팀 기준에 따라 유동적으로 변경 가능

→ Jira 컨벤션(Rule) 정하기 

**Jira 컨벤션**

- 명명 규칙을 일관성 있게 작성
- 내용 설명 : “무엇을 왜 하는지” 설명 → 담당자에게 물어볼 필요 X
- 완료 조건 : 명확한 완료 조건 → QA나 리뷰, 회고시 검증 가능

**스토리 포인트**

- 작업 구현에 필요한 전반적인 노력의 추정치
    
    ⇒ 추정치이기 때문에 개발 전 상세한 예측이 필요하고, 준비가 구체적일수록 리스크 관리에 유리
    
- KPT 회고를 통해 지속적 개선
- 1 Point ≠ 1시간
- 팀원 모두가 공감할 수 있는 기준
- 2-3포인트가 베이스라면 8포인트 이상의 작업은 분할

**번다운 차트**

남은 작업량을 기반으로 스프린트 목표 달성 가능성을 예측하고 진행 상황을 관리

- 추정 통계 : 사용자가 선택한 추정 통계를 나타냄
- 남은 값 : 스프린트에 남은 총 작업량
- 안내성 : 팀이 선형으로 진행된다고 가정할 때 대략적인 위치
